# 执行工作区和运行时服务

> **核心摘要**: 本指南记录 Paperclip 中项目运行时配置、执行工作区和 Issue 运行的预期运行时模型。

## 概述

Paperclip 现在将此呈现为工作区-命令模型：
- `Services` 是保持监督的长运行命令
- `Jobs` 是运行一次并退出的单次命令
- 原始运行时 JSON 仍可用于高级配置，但不再是主要心智模型

## 项目运行时配置

你可以在项目工作区本身上定义如何运行项目：
- 项目工作区运行时配置描述该项目的可用服务和作业
- 这是子执行工作区可能继承的默认运行时配置
- 定义配置本身不会启动任何内容

## 手动运行时控制

工作区命令从 UI 手动控制：
- 项目工作区服务从项目工作区 UI 启动和停止
- 执行工作区服务从执行工作区 UI 启动和停止
- Paperclip 不会作为 Issue 执行的一部分自动启动或停止这些工作区服务
- Paperclip 也不会在服务器启动时自动重启工作区服务

## Issue 和执行工作区

Issue 附加到执行工作区行为，而不是自动运行时管理：
- Issue 可以选择隔离工作区模式创建新执行工作区
- Issue 可以在选择 reuse 时重用现有执行工作区
- 多个 Issue 可以故意共享一个执行工作区，以便它们可以在同一分支和运行中的运行时服务上工作

## 执行工作区生命周期

执行工作区持久存在直到人类关闭它们：
- UI 可以归档执行工作区
- 关闭执行工作区停止其运行时服务并在允许时清理其工作区工件
- 指向项目主 checkout 的共享工作区在清理期间比一次性隔离工作区更保守处理

## Heartbeat 运行期间的工作区解析

Heartbeat 仍解析运行的工作区，但那关于代码位置和会话连续性，不是运行时服务控制。

当前实现保证：
- 项目工作区命令配置是执行工作区 UI 控制的回退
- 执行工作区运行时覆盖存储在执行工作区上
- Heartbeat 运行不会自动启动工作区服务
- 服务器启动不会自动重启工作区服务
