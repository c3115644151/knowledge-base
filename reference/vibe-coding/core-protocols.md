# Vibe Coding 核心行为协议 (IF-THEN Rules)

> **协议类型**：强制执行规则
> **适用范围**：所有 Vibe Coding 任务
> **更新版本**：v1.0

---

## 一、AI-人类协作协议

### 1.1 角色分工规则

```
IF 人类提出需求 THEN
  → AI 负责理解、执行、生成产出
  → 人类负责描述意图、验证结果、做决策
END IF

IF AI 生成代码 THEN
  → 必须包含输入输出示例
  → 必须说明可能失败的情况
  → 必须标注需要人类确认的决策点
END IF

IF 人类验证失败 THEN
  → AI 必须等待人类反馈
  → 不得擅自修改未经验证的代码
END IF
```

### 1.2 上下文管理规则

```
IF 开始新任务 THEN
  → 优先检查 memory-bank 目录
  → 读取 @architecture.md 了解项目结构
  → 读取 @progress.md 了解当前进度
END IF

IF 上下文超过 Token 限制 THEN
  → 使用 /clear 或 /compact 清理
  → 保留关键架构和进度信息
END IF

IF 需要回顾历史决策 THEN
  → 参考 memory-bank/@progress.md
  → 参考 memory-bank/@architecture.md
END IF
```

### 1.3 规划驱动规则

```
IF 人类要求实现功能 THEN
  → AI 必须先输出实施计划
  → 等待人类确认后再执行
  → 每完成一步报告进度
END IF

IF 人类要求快速实现 THEN
  → AI 仍需输出最小计划（3-5步）
  → 包含验证步骤
  → 确保每步可独立验证
END IF
```

## 二、代码生成协议

### 2.1 模块化规则

```
IF 生成新代码 THEN
  → 必须拆分到多个文件（禁止 monolith）
  → 每个文件职责单一
  → 文件名必须体现功能
END IF

IF 代码超过 500 行 THEN
  → 必须拆分模块
  → 优先按功能拆分
END IF

IF 多个功能相关 THEN
  → 放入同一目录
  → 创建 __init__.py 导出公共接口
END IF
```

### 2.2 依赖管理规则

```
IF 需要引入新依赖 THEN
  → 优先使用标准库
  → 其次使用成熟开源库（生产验证）
  → 最后考虑自研
END IF

IF 自研 vs 复用存在冲突 THEN
  → 优先复用（胶水编程原则）
  → 复用成本明显更低时必须复用
END IF

IF 引入新库 THEN
  → 必须更新 requirements.txt
  → 必须说明为什么现有库不满足
END IF
```

### 2.3 错误处理规则

```
IF 函数可能失败 THEN
  → 必须返回错误信息或使用异常
  → 不得返回 None 表示失败（除非显式）
  → 必须说明失败原因
END IF

IF 外部依赖可能超时 THEN
  → 必须设置超时
  → 必须说明超时处理策略
END IF

IF 异常被捕获 THEN
  → 必须记录日志
  → 必须向上传播或显式处理
  → 不得吞掉异常
END IF
```

## 三、胶水编程协议（核心）

### 3.1 零幻觉规则

```
IF 功能已存在于成熟开源库 THEN
  → 必须直接复用
  → 禁止自行实现相同功能
  → 禁止对成熟库进行功能裁剪
END IF

IF 寻找实现方案 THEN
  → 优先搜索 GitHub Topics
  → 优先选择 Star 数 > 10k 的项目
  → 优先选择有持续维护的项目
END IF

IF 依赖路径配置 THEN
  → 路径必须真实指向完整源码
  → 禁止使用简化版/替代版
  → 禁止路径遮蔽导致加载错误实现
END IF
```

### 3.2 胶水层职责规则

```
IF 当前项目需要代码 THEN
  → 胶水层职责：组合、调用、封装、适配
  → 禁止在胶水层实现核心算法
  → 禁止复制依赖库代码到项目
END IF

IF 需要修改依赖库行为 THEN
  → 必须通过配置或扩展点实现
  → 禁止直接修改依赖库源码
END IF
```

## 四、质量保障协议

### 4.1 测试验证规则

```
IF 完成功能实现 THEN
  → 必须编写测试用例
  → 测试必须覆盖正常和异常路径
  → 必须运行测试验证
END IF

IF 测试失败 THEN
  → 不得跳过或忽略
  → 必须修复或明确标注待修复
END IF

IF 没有测试覆盖 THEN
  → 必须说明原因
  → 必须标注为高风险区域
END IF
```

### 4.2 代码审查规则

```
IF 提交代码变更 THEN
  → 自检：是否引入新问题
  → 检查：是否违反 SOLID/DRY 原则
  → 确认：是否有未使用的变量/函数
END IF

IF 发现代码问题 THEN
  → 使用现象学还原方法
  → 描述：现象/意图/情境/边界
  → 输出最小可复现示例
END IF
```

### 4.3 性能考虑规则

```
IF 代码可能存在性能问题 THEN
  → 必须标注性能敏感区域
  → 必须说明瓶颈所在
  → 必须给出优化方向
END IF

IF 涉及并发/异步 THEN
  → 必须考虑竞态条件
  → 必须考虑资源泄漏
  → 必须考虑超时处理
END IF
```

## 五、安全协议

### 5.1 凭证管理规则

```
IF 需要使用密钥/Token THEN
  → 必须从环境变量读取
  → 禁止硬编码在代码中
  → 禁止提交到 Git
END IF

IF 发现硬编码凭证 THEN
  → 必须标记为安全问题
  → 必须建议使用环境变量替代
END IF
```

### 5.2 输入验证规则

```
IF 接收外部输入 THEN
  → 必须进行类型验证
  → 必须进行边界验证
  → 必须防止注入攻击
END IF

IF 处理用户数据 THEN
  → 必须考虑数据脱敏
  → 必须考虑日志脱敏
END IF
```

## 六、架构决策协议

### 6.1 可逆性优先规则

```
IF 存在多种实现方案 THEN
  → 优先选择可逆性高的方案
  → 标注：哪些决策是不可逆的
  → 不可逆决策需要人类确认
END IF

IF 需要绑定外部系统 THEN
  → 先定义接口/抽象
  → 使用适配器模式
  → 延后绑定具体实现
END IF
```

### 6.2 复杂度控制规则

```
IF 引入新抽象/框架 THEN
  → 必须说明必要性
  → 必须说明复杂度增加
  → 必须对比简化方案
END IF

IF 代码复杂度超过阈值 THEN
  → 必须重构
  → 优先拆分而非优化
END IF
```

## 七、异常处理决策树

```
START
├── IF 任务不清晰
│   └── 执行现象学还原：现象/意图/情境/边界
│       → 输出 MRE（最小可复现示例）
│
├── IF 代码不工作
│   ├── IF 有错误信息
│   │   └── 完整复制错误给 AI
│   ├── IF 无错误但行为异常
│   │   └── 提供输入/输出/预期对比
│   └── IF 性能问题
│       └── 提供 benchmark 数据
│
├── IF AI 陷入循环
│   └── 换个思路描述问题
│       → 开新对话
│       → 简化问题描述
│
└── IF 需要新功能
    └── 先问 AI 推荐方案
        → 人类决策后执行
```

## 八、版本控制协议

```
IF 完成可验证的步骤 THEN
  → 提交 Git
  → 更新 progress.md
  → 更新 architecture.md（如有必要）
END IF

IF 提交前检查
  → git status 确认改动范围
  → git diff 确认改动内容
  → 编写有意义的 commit message
END IF

IF 需要回退 THEN
  → Claude Code: /rewind
  → 其他工具: git reset
END IF
```

## 九、强制约束清单

以下规则**绝对不可违反**：

1. **禁止**：生成不存在的 API 调用
2. **禁止**：修改生产环境数据（未确认）
3. **禁止**：提交硬编码凭证
4. **禁止**：跳过测试验证
5. **禁止**：忽略错误信息
6. **禁止**：删除未备份的重要文件
7. **禁止**：在不了解的情况下修改架构
8. **禁止**：引入未验证安全性的依赖

## 十、协议触发检查清单

每次任务开始时，AI 必须确认：

- [ ] 已读取 memory-bank 目录（如存在）
- [ ] 已理解项目架构
- [ ] 已确认任务目标
- [ ] 已识别约束条件
- [ ] 已规划实施步骤
- [ ] 已标注需要人类决策的点
