# 胶水编程原理 (Glue Coding Principles)

> **原则类型**：核心工程原则
> **核心口号**：能抄不写，能连不造，能复用不原创
> **AI 执行要点**：优先复用成熟组件，最小化自研代码

---

## 一、范式转移

```
传统编程：人写代码
Vibe Coding：AI 写代码，人审代码
胶水编程：AI 连接代码，人审连接

核心转变：
├─ 从「生成」到「连接」
├─ 从「造轮子」到「选轮子」
└─ 从「写代码」到「描述意图」
```

---

## 二、零幻觉规则

### 2.1 核心原则

```
IF 功能已存在于成熟开源库 THEN
  → 必须直接复用
  → 禁止自行实现相同功能
  → 禁止对成熟库进行功能裁剪
END IF

IF 寻找实现方案 THEN
  → 优先搜索 GitHub Topics
  → 优先选择 Star 数 > 10k 的项目
  → 优先选择有持续维护的项目
END IF
```

### 2.2 GitHub 找轮子策略

| 策略 | 描述 |
|:---|:---|
| GitHub Topics | 访问 https://github.com/topics/xxx |
| 搜索关键词 | 根据需求搜索相关仓库 |
| 排序筛选 | 按 Star、Fork、更新时间筛选 |
| 组合 Topic | `https://github.com/topics/python?q=telegram` |

| 需求类型 | 推荐 Topic |
|:---|:---|
| Telegram Bot | telegram-bot |
| 数据分析 | data-analysis |
| AI Agent | ai-agent |
| CLI 工具 | cli |
| Web 爬虫 | web-scraping |

## 三、架构哲学

```
┌─────────────────────────────────────────────────────┐
│                    业务需求                          │
└─────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────┐
│                   AI 胶水层 (Glue Layer)              │
│                                                     │
│   "我理解你要做什么，让我把这些积木连起来"                  │
│                                                     │
└─────────────────────────────────────────────────────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
   │  成熟模块 A  │  │  成熟模块 B  │  │  成熟模块 C  │
   │  (10万+ ⭐)  │  │  (生产验证)  │  │  (官方 SDK)  │
   └─────────────┘  └─────────────┘  └─────────────┘

胶水分层：
├─ 实体 (Entity)：成熟的开源项目、官方 SDK、久经考验的库
├─ 连接 (Link)：AI 生成的胶水代码，负责数据流转和接口适配
└─ 功能 (Function)：你描述的业务目标
```

---

## 四、胶水层职责

### 4.1 允许的职责

```
胶水层只承担：
├─ 组合 (composition)
├─ 调用 (invocation)
├─ 封装 (wrapper)
└─ 适配 (adapter)
```

### 4.2 禁止的职责

```
胶水层禁止：
├─ 实现核心算法
├─ 实现数据结构
├─ 实现通用逻辑
├─ 复制依赖库代码
└─ 修改依赖库源码
```

## 五、标准流程

```
FUNCTION glue_coding_workflow(业务需求)
  │
  ├─► 1. 明确目标
  │     └─► "我要实现 XXX 功能"
  │
  ├─► 2. 寻找轮子
  │     ├─► "有没有成熟的库/项目已经做过类似的事？"
  │     ├─► 使用 GitHub Topics 搜索
  │     └─► 评估 Star 数、维护状态、许可证
  │
  ├─► 3. 理解接口
  │     ├─► 阅读官方文档
  │     └─► 总结：输入是什么，输出是什么
  │
  ├─► 4. 描述连接
  │     ├─► "A 的输出要变成 B 的输入"
  │     └─► AI 生成胶水代码
  │
  └─► 5. 验证运行
        ├─► IF 跑通 → 完成
        └─► IF 报错 → 把错误扔给 AI，继续粘

END FUNCTION
```

## 六、胶水编程 vs 传统开发

| 维度 | 传统开发 | 胶水编程 |
|:---|:---|:---|
| 功能实现方式 | 自己写 | 复用开源 |
| 工作量 | 大 | 小得多 |
| 成功率 | 不确定 | 高 |
| 速度 | 慢 | 极快 |
| 错误率 | 容易踩坑 | 使用成熟方案 |
| 重点 | "造轮子" | "组合轮子" |
| 幻觉风险 | 高 | 零（使用成熟代码） |
| 复杂度 | 自研承担 | 转嫁给社区 |

## 七、经典案例

### 案例：Polymarket 数据分析 Bot

```
需求：实时获取 Polymarket 数据，分析后推送到 Telegram

传统做法：
├─ 从零写爬虫 → 500 行代码
├─ 写分析逻辑 → 300 行代码
├─ 写 Bot → 200 行代码
├─ 总计：1000+ 行代码
└─ 开发时间：2 周

胶水做法：
├─ 轮子 1: polymarket-py (官方 SDK)
├─ 轮子 2: pandas (数据分析)
├─ 轮子 3: python-telegram-bot (消息推送)
├─ 胶水代码: 50 行
└─ 开发时间: 2 小时
```

## 八、反模式

```
以下模式违反胶水编程原则：

❌ 自行实现已有成熟库的功能
❌ 复制依赖库代码到项目再修改
❌ 对依赖库进行功能裁剪
❌ 使用简化版替代生产级库
❌ 路径遮蔽导致加载错误实现
❌ 在胶水层实现核心算法
❌ 创建"先写接口后补实现"的占位符
```

## 九、质量检查

```
胶水编程质量检查：

IF 引入新依赖 THEN
  ├─► Star 数 > 10k？
  ├─► 近期有更新（3个月内）？
  ├─► 官方维护或有活跃社区？
  ├─► 许可证允许商业使用？
  ├─► 文档完整？
  └─► IF 全部满足 → 可以使用
END IF

胶水代码检查：
├─► 代码行数是否最小？
├─► 是否只是组合/适配逻辑？
├─► 是否包含核心算法？
└─► IF 包含核心算法 → 违反胶水原则
```
