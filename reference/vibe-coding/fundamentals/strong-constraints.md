# 强前置条件约束 (Strong Constraints)

> **约束类型**：绝对不可违反的规则
> **总数**：97 条强制规则
> **分类**：通用开发约束、胶水开发约束、完整性检查约束
> **AI 执行要点**：代码审查和架构评审时强制执行

---

## 一、通用开发约束 (34 条)

### 1.1 代码组织

| ID | 约束规则 |
|:---|:---|
| GC-01 | 不得采用只解决局部问题的补丁式修改而忽视整体设计与全局优化 |
| GC-02 | 不得引入过多用于中间通信的中间状态以免降低可读性并形成循环依赖 |
| GC-03 | 不得为过渡场景编写大量防御性代码以免掩盖主逻辑并增加维护成本 |
| GC-04 | 不得只追求功能完成而忽略架构设计 |
| GC-05 | 不得省略必要注释，代码必须对他人和未来维护者可理解 |
| GC-06 | 不得编写难以阅读的代码，必须保持结构简单清晰并添加解释性注释 |
| GC-07 | 不得违反 SOLID 与 DRY 原则，必须保持职责单一并避免逻辑重复 |
| GC-08 | 不得维护复杂的中间状态，仅允许保留最小必要的核心数据 |

### 1.2 状态管理

| ID | 约束规则 |
|:---|:---|
| GC-09 | 不得依赖外部或临时中间状态驱动 UI，所有 UI 状态必须从核心数据推导 |
| GC-10 | 不得通过隐式或间接方式变更状态，状态变化应直接更新数据并由框架重新计算 |
| GC-11 | 不得编写过量的防御性代码，应通过清晰的数据约束与边界设计解决问题 |
| GC-12 | 不得保留未被使用的变量和函数 |
| GC-13 | 不得将状态提升或集中到不必要的层级，状态应在最接近使用的位置管理 |

### 1.3 依赖与架构

| ID | 约束规则 |
|:---|:---|
| GC-14 | 不得在业务代码中直接依赖具体实现细节或硬编码外部服务 |
| GC-15 | 不得在核心业务逻辑中混入 IO、网络、数据库等副作用操作 |
| GC-16 | 不得形成隐式依赖，如依赖调用顺序、全局初始化或副作用时序 |
| GC-17 | 不得吞掉异常或使用空 catch 掩盖错误 |
| GC-18 | 不得将异常作为正常控制流的一部分 |
| GC-19 | 不得返回语义不清或混用的错误结果（如 null / undefined / false） |

### 1.4 数据管理

| ID | 约束规则 |
|:---|:---|
| GC-20 | 不得在多个位置同时维护同一份事实数据 |
| GC-21 | 不得在未定义生命周期和失效策略的情况下缓存状态 |
| GC-22 | 不得跨请求共享可变状态，除非明确设计为并发安全 |

### 1.5 命名与职责

| ID | 约束规则 |
|:---|:---|
| GC-23 | 不得使用语义模糊或误导性的命名 |
| GC-24 | 不得让单个函数或模块承担多个不相关语义 |
| GC-25 | 不得引入非必要的时间耦合或隐含时间假设 |

### 1.6 复杂度控制

| ID | 约束规则 |
|:---|:---|
| GC-26 | 不得在关键路径中引入不可控的复杂度或隐式状态机 |

### 1.7 调研与确认

| ID | 约束规则 |
|:---|:---|
| GC-27 | 不得臆测接口行为，必须先查询文档、定义或源码 |
| GC-28 | 不得在需求、边界或输入输出不清晰的情况下直接实现 |
| GC-29 | 不得基于猜测实现业务逻辑，必须与人类确认需求并留痕 |
| GC-30 | 不得在未评估现有实现的情况下新增接口或模块 |

### 1.8 质量保证

| ID | 约束规则 |
|:---|:---|
| GC-31 | 不得跳过验证流程，必须编写并执行测试用例 |
| GC-32 | 不得触碰架构红线或绕过既有设计规范 |
| GC-33 | 不得假装理解需求或技术细节，不清楚时必须明确说明 |
| GC-34 | 不得在缺乏上下文理解的情况下直接修改代码，必须基于整体结构审慎重构 |

## 二、胶水开发约束 (23 条)

### 2.1 复用原则

| ID | 约束规则 |
|:---|:---|
| GL-01 | 不得自行实现底层或通用逻辑，必须优先、直接、完整复用既有成熟仓库与生产级库 |
| GL-02 | 不得为了方便而复制依赖库代码到当前项目中再修改使用 |
| GL-03 | 不得对依赖库进行任何形式的功能裁剪、逻辑重写或降级封装 |
| GL-04 | 允许使用本地源码直连或包管理器安装方式，但实际加载的必须是完整生产级实现 |
| GL-05 | 不得使用简化版、替代版或重写版依赖冒充真实库实现 |

### 2.2 依赖真实性

| ID | 约束规则 |
|:---|:---|
| GL-06 | 所有依赖路径必须真实存在并指向完整仓库源码 |
| GL-07 | 不得通过路径遮蔽、重名模块或隐式 fallback 加载非目标实现 |
| GL-08 | 代码中必须直接导入完整依赖模块，不得进行子集封装或二次抽象 |
| GL-09 | 不得在当前项目中实现依赖库已提供的同类功能 |
| GL-10 | 所有被调用能力必须来自依赖库的真实实现，不得使用 Mock、Stub 或 Demo 代码 |

### 2.3 实现完整性

| ID | 约束规则 |
|:---|:---|
| GL-11 | 不得存在占位实现、空逻辑或"先写接口后补实现"的情况 |

### 2.4 胶水层职责

| ID | 约束规则 |
|:---|:---|
| GL-12 | 当前项目仅允许承担业务流程编排、模块组合调度、参数配置与输入输出适配职责 |
| GL-13 | 不得在当前项目中重复实现算法、数据结构或复杂核心逻辑 |
| GL-14 | 不得将依赖库中的复杂逻辑拆出后自行实现 |

### 2.5 运行验证

| ID | 约束规则 |
|:---|:---|
| GL-15 | 所有导入的模块必须在运行期真实参与执行 |
| GL-16 | 不得存在"只导入不用"的伪集成行为 |
| GL-17 | 必须确保 sys.path 或依赖注入链路加载的是目标生产级本地库 |
| GL-18 | 不得因路径配置错误导致加载到裁剪版、测试版或简化实现 |

### 2.6 透明性

| ID | 约束规则 |
|:---|:---|
| GL-19 | 在生成代码时必须明确标注哪些功能来自外部依赖 |
| GL-20 | 在任何情况下不得生成或补写依赖库内部实现代码 |

### 2.7 代码量原则

| ID | 约束规则 |
|:---|:---|
| GL-21 | 只允许生成最小必要的胶水代码与业务层调度逻辑 |
| GL-22 | 必须假设依赖库为权威且不可修改的黑箱实现 |

### 2.8 评价标准

| ID | 约束规则 |
|:---|:---|
| GL-23 | 项目评价标准以是否正确、完整站在成熟系统之上构建为唯一依据，而非代码量 |

## 三、完整性检查约束 (40 条)

### 3.1 功能完整性

| ID | 约束规则 |
|:---|:---|
| IC-01 | 不得允许任何形式的功能弱化、裁剪或替代实现通过审计 |
| IC-02 | 必须确认所有功能模块均为完整生产级实现 |

### 3.2 实现真实性

| ID | 约束规则 |
|:---|:---|
| IC-03 | 不得存在阉割逻辑、Mock、Stub 或 Demo 级替代代码 |
| IC-04 | 必须确保行为与生产环境成熟版本完全一致 |
| IC-05 | 必须验证当前工程是否 100% 复用既有成熟代码 |

### 3.3 复用完整性

| ID | 约束规则 |
|:---|:---|
| IC-06 | 不得存在任何形式的重新实现或功能折叠 |
| IC-07 | 必须确认当前工程为直接集成而非复制后修改 |

### 3.4 路径验证

| ID | 约束规则 |
|:---|:---|
| IC-08 | 必须核查所有本地库导入路径真实、完整且生效 |
| IC-09 | 必须确认 datas 模块为完整数据模块而非子集 |
| IC-10 | 必须确认 sizi.summarys 为完整算法实现且未降级 |

### 3.5 行为一致性

| ID | 约束规则 |
|:---|:---|
| IC-11 | 不得允许参数简化、逻辑跳过或隐式行为改变 |

### 3.6 执行验证

| ID | 约束规则 |
|:---|:---|
| IC-12 | 必须确认所有导入模块在运行期真实参与执行 |

### 3.7 接口验证

| ID | 约束规则 |
|:---|:---|
| IC-13 | 不得存在接口空实现或导入不调用的伪集成 |

### 3.8 路径遮蔽检查

| ID | 约束规则 |
|:---|:---|
| IC-14 | 必须检查并排除路径遮蔽、重名模块误导加载问题 |

### 3.9 审计原则

| ID | 约束规则 |
|:---|:---|
| IC-15 | 所有审计结论必须基于可验证的代码与路径分析 |
| IC-16 | 不得输出模糊判断或基于主观推测的结论 |
| IC-17 | 审计输出必须明确给出结论、逐项判断及风险后果 |

## 四、约束执行检查清单

### 4.1 代码提交前检查

```
[ ] 未使用未声明的变量
[ ] 无空 catch 块
[ ] 无硬编码凭证
[ ] 依赖路径真实有效
[ ] 模块职责单一
[ ] 异常被正确处理
[ ] 状态变更显式
[ ] 遵循 SOLID/DRY
```

### 4.2 胶水代码检查

```
[ ] 未自行实现已有库的功能
[ ] 未复制库代码到项目
[ ] 未对库进行功能裁剪
[ ] 依赖为生产级完整实现
[ ] 代码为最小必要胶水
[ ] 导入模块运行期真实执行
[ ] 未使用 Mock/Stub 替代
```

### 4.3 功能完整性检查

```
[ ] 所有声明的功能已实现
[ ] 无 Demo/Stub 占位
[ ] 无功能降级或裁剪
[ ] 行为与成熟版本一致
```

## 五、约束违规处理

```
IF 发现约束违规 THEN
  ├─► 立即标记为问题
  ├─► 说明违反的约束编号
  ├─► 提供修复建议
  └─► 不得继续执行违规代码
END IF

IF 违反安全相关约束 THEN
  ├─► 立即停止操作
  ├─► 标记为安全问题
  └─► 必须修复后才能继续
END IF
```
