# 全自动开发闭环工作流

> 基于状态机 + 文件 Hook 的五步 AI Agent 工作流系统

## 执行协议

### 触发条件

| 场景 | 触发方式 |
|:---|:---|
| 新功能开发 | 提供原始需求，启动 Step1 规格锁定 |
| 自动化流水线 | CI/CD 触发 runner.py |
| 持续集成 | 文件变更触发 Hook 自动执行 |
| 循环迭代 | Step5 判定失败时自动回跳 Step2 |

### 输入输出规范

#### 状态文件 Schema
```json
{
  "run_id": "20251225T054501",
  "step": "step3",
  "status": "pending|running|success|failed|completed|fatal_error",
  "payload_path": "artifacts/<run>/step2.json",
  "verify": {"status": "success|failed", "details": "..."},
  "target_step": "step2|step5|done",
  "retry_count": 0
}
```

#### 各步骤产物

| 步骤 | 产物 | 说明 |
|:---|:---|:---|
| Step1 | `locked_spec.md` | 《锁定规格书》 |
| Step2 | `execution_plan.md` | 《综合执行方案》+ DAG + Gantt |
| Step3 | `changeset/` | 代码变更集 + 决策日志 |
| Step4 | `evidence_package/` | 测试报告 + 安全扫描 + 裁定结果 |
| Step5 | `final_report/` | 成功报告 或 回跳上下文 |

### 核心步骤详解

#### Step 1: 需求输入 (规格锁定 Agent)
```
角色: 需求架构师 + 系统设计顾问
输入: 用户原始需求 (模糊/非结构化)
处理:
  Phase 1: 理解与意图发散
    - 解构显性目标、隐性动机
    - 生成 2-3 个需求解读草案
  Phase 2: 结构化澄清
    - 围绕目标、范围、验收、约束提问
  Phase 3: 规格形式化
    - 生成《锁定规格书》草稿
    - 用户确认
输出: 《锁定规格书》(Locked Specification)
检查点: 用户明确回复"确认"
```

#### Step 2: 执行计划 (计划编排 Agent)
```
角色: AI 技术主管 + 系统架构师
输入: 《锁定规格书》
处理:
  Phase 1: 规格解析与架构映射
    - 构建最小可行技术架构
    - 声明架构假设
  Phase 2: 任务分层 (DAG)
    - 3层结构: 里程碑 → 模块 → 任务
    - 生成 Gantt 图 + 依赖图
  Phase 3: 正交计划
    - 测试计划 (AC → 测试用例)
    - 回滚预案 (SOP)
    - 监控告警计划 (KPIs + 阈值)
输出: 《综合执行方案》
检查点: 包含任务DAG、测试、回滚、监控四个完整部分
```

#### Step 3: 实施变更 (实施变更 Agent)
```
角色: 原则驱动的 AI 软件工程师
输入: 《综合执行方案》
处理:
  Phase 1: 初始化与环境校验
  Phase 2: 任务处理循环
    - 按 DAG 顺序执行任务
    - 生成/修改代码
    - 遵循 KISS/DRY/SOLID 原则
    - 记录决策日志
  Phase 3: 版本控制
    - 模块完成即 git commit
  Phase 4: 产物聚合
    - 生成最终 changeset
    - 整理决策日志
输出: 代码变更集 + 实施日志
检查点: 100% 忠于计划，无偏离
```

#### Step 4: 验证发布 (验证发布 Agent)
```
角色: 自动化 QA + 发布守门员
输入: 《综合执行方案》+ 《变更集》+ 《实施日志》
处理:
  Phase 1: 输入校验与关联
  Phase 2: 自动化验证执行
    - 功能测试 (单元/集成/E2E)
    - 安全扫描 (SAST/DAST)
    - 代码质量审计
  Phase 3: 证据包聚合与裁定
    - GO: 执行发布 + 建立监控基线
    - NO-GO: 执行回滚 + 记录失败原因
输出: 《验证与发布证据包》+ 裁定结果
检查点: P0失败或S0漏洞立即终止
```

#### Step 5: 总控与循环 (总控循环 Agent)
```
角色: AI 项目总指挥 + 系统控制 Agent
输入: 《验证与发布证据包》
处理:
  Phase 1: 状态评估
    - 读取主任务清单
    - 更新任务状态
  Phase 2: 决策网关
    - GO → 成功工作流
    - NO-GO → 失败工作流
  Phase 3: 工作流执行
    - 成功: 归档 + 下一个任务
    - 失败: 根因分析 + 准备重规划上下文
  Phase 4: 循环控制
    - 有待处理 → 重新调用 Step2
    - 全部完成 → 流程结束
输出: 成功报告 或 回跳信号
检查点: 所有任务"完美完成"才能结束
```

### 检查点清单

| 步骤 | 检查项 | 标准 | 修复措施 |
|:---|:---|:---|:---|
| Step1 | 用户确认 | 用户明确回复"确认" | 继续澄清 |
| Step2 | 计划完备性 | DAG+测试+回滚+监控齐全 | 补充缺失部分 |
| Step3 | 计划保真度 | 100% 忠于计划 | 重做或人工审查 |
| Step4 | 证据充分性 | 所有 AC 有测试覆盖 | 补充测试 |
| Step4 | P0/S0 | 致命问题为零 | 立即回滚 |
| Step5 | 任务完成 | 所有任务 ✅ | 回跳 Step2 |

### 文件约定

| 类型 | 路径 | 说明 |
|:---|:---|:---|
| 状态文件 | `workflow_engine/state/current_step.json` | 唯一调度入口 |
| 产物目录 | `workflow_engine/artifacts/<run_id>/` | 每次运行产物 |
| Hook 脚本 | `workflow_engine/hook_runner.sh` | inotify 文件监听 |
| Runner | `workflow_engine/runner.py` | 状态机调度器 |

### 运行方式

```bash
# 方式1: Kiro CLI
kiro-cli chat --agent workflow

# 方式2: 手动运行
python3 workflow_engine/runner.py start

# 方式3: Hook 自动模式
# 终端1: 启动 Hook
./workflow_engine/hook_runner.sh
# 终端2: 触发工作流
python3 workflow_engine/runner.py start
```
