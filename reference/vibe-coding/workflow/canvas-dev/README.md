# Canvas 白板驱动开发工作流

> 图形是第一公民，代码是白板的序列化形式

## 执行协议

### 触发条件

| 场景 | 触发方式 |
|:---|:---|
| 新项目理解 | 提供项目代码路径，AI 扫描生成架构白板 |
| 新功能开发 | 在白板绘制模块和依赖关系，AI 生成代码 |
| 架构重构 | 修改白板连线，AI 同步重构代码 |
| Code Review | AI 对比白板与代码，发现异常调用链 |
| 团队协作 | 新人入职、技术评审、需求变更讨论 |

### 输入输出规范

#### 输入
| 类型 | 说明 | 必需 |
|:---|:---|:---|
| 项目代码路径 | 待分析或开发的源代码目录 | 是 |
| Canvas 模板 | 可选的项目/模块白板模板 | 否 |
| 分析粒度 | file/class/service 三选一 | 否 |

#### 输出
| 类型 | 说明 | 格式 |
|:---|:---|:---|
| 架构白板 | 模块依赖可视化 | `.canvas` (JSON) |
| 代码变更 | 基于白板生成的代码 | 源代码文件 |
| 同步报告 | 白板与代码一致性检查结果 | Markdown |

### 核心步骤详解

#### Step 1: 架构分析
```
输入: 项目代码路径 + 粒度要求
动作: 
  1. AST 解析源代码结构
  2. 提取 import/require 依赖关系
  3. 识别入口文件和核心模块
  4. 生成 Canvas JSON (节点+连线+分组)
输出: .canvas 文件
检查点: 节点覆盖核心模块，连线反映真实依赖
```

#### Step 2: 白板驱动编码
```
输入: Canvas JSON + 技术栈 + 目标目录
动作:
  1. 解析节点 → 文件/类定义
  2. 解析连线 → import/调用关系
  3. 按分组生成目录结构
  4. 生成完整代码文件
输出: 代码变更集
检查点: 文件结构与白板一致，调用关系正确
```

#### Step 3: 同步检查
```
输入: Canvas JSON + 项目代码
动作:
  1. 对比节点与实际文件
  2. 验证连线与调用关系
  3. 检查分组与目录一致性
输出: 一致性报告 (严重/轻微/良好)
检查点: 覆盖率 ≥ 90%, 准确率 ≥ 95%
```

### 检查点清单

| 检查项 | 标准 | 修复措施 |
|:---|:---|:---|
| 节点完整性 | 所有代码文件有对应节点 | 添加缺失节点 |
| 连线准确性 | 连线反映真实调用关系 | 修正/删除错误连线 |
| 分组一致性 | 分组与目录结构匹配 | 重新分组 |
| 命名统一 | 节点名与文件名一致 | 重命名对齐 |
