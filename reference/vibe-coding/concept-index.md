# Vibe Coding 概念索引 (Concept Index)

> **索引类型**：定义|触发|指引
> **用途**：AI 快速检索和理解核心概念
> **格式**：概念名 → 定义 | 触发条件 | 行为指引

---

## 一、核心概念

### Vibe Coding

| 属性 | 内容 |
|:---|:---|
| **定义** | 以自然语言驱动、让 LLM 生成大部分代码的开发方式 |
| **核心特征** | 先沉浸式做出能跑的东西，极低门槛快速产出原型 |
| **触发条件** | 当人类需要快速将想法变成可运行代码时 |
| **行为指引** | 人类描述意图，AI 生成代码，循环迭代直到满意 |
| **风险** | 可控性与可靠性风险，需要规划驱动和模块化约束 |

### 胶水编程 (Glue Coding)

| 属性 | 内容 |
|:---|:---|
| **定义** | 复用成熟开源组件，通过最小量胶水代码组合成完整系统 |
| **核心理念** | 能抄不写，能连不造，能复用不原创 |
| **触发条件** | 当需要实现新功能时，优先搜索现有成熟解决方案 |
| **行为指引** | 寻找成熟库 → 理解接口 → 编写胶水层连接 |
| **价值** | 零幻觉、复杂性归零、门槛消失 |

### Memory Bank

| 属性 | 内容 |
|:---|:---|
| **定义** | 项目根目录下的上下文持久化目录 |
| **包含文件** | architecture.md, progress.md, game-design-document.md, tech-stack.md |
| **触发条件** | 每次新对话或任务切换时 |
| **行为指引** | 先读取 → 理解架构 → 继续之前进度 |

## 二、方法论概念

### 现象学还原

| 属性 | 内容 |
|:---|:---|
| **定义** | 先把"解释/偏好"清零，回到可观察事实与可复现结构 |
| **触发条件** | 需求含糊、模型脑补、Bug 难复现时 |
| **行为指引** | 输出四件套：现象/意图/情境/边界/未确定项/MRE |
| **工具** | pytest（MRE 脚本）、日志、最小数据样例 |

### 正反合迭代

| 属性 | 内容 |
|:---|:---|
| **定义** | 三轮迭代：正（快速可用）→ 反（反例打脸）→ 合（收敛为工程版本） |
| **触发条件** | 避免一次性"写到完美" |
| **行为指引** | 先只做 main path → 列失败模式 → 重构收敛 |
| **工具** | pytest + hypothesis + ruff/black + profiling |

### 可证伪主义

| 属性 | 内容 |
|:---|:---|
| **定义** | 把"看起来对"变成"暂时无法证伪" |
| **触发条件** | 显著降低隐藏 bug |
| **行为指引** | 每个关键断言配一个能让它失败的测试 |
| **工具** | hypothesis（性质/模糊测试）、pytest |

### 奥卡姆剃刀

| 属性 | 内容 |
|:---|
| **定义** | 在满足全部测试与验收的前提下，删减实现复杂度 |
| **触发条件** | 避免模型引入不必要框架/抽象 |
| **行为指引** | 要求两套方案对比；以测试为准删复杂度 |
| **工具** | ruff（复杂度/风格）、依赖审计 |

### 溯因推理 (Abduction)

| 属性 | 内容 |
|:---|:---|
| **定义** | 比穷举更快定位"最可能原因"的推理方式 |
| **触发条件** | debug/性能退化时 |
| **行为指引** | 列候选原因 → 为每个写最便宜的区分性实验 |
| **工具** | 结构化日志、trace、最小 benchmark、feature flag |

## 三、语言层概念 (L1-L12)

### L1: 控制语法
| 属性 | 内容 |
|:---|:---|
| **定义** | 变量、if/else、for/while、函数/return 等基础语法 |
| **触发条件** | 编写简单脚本或理解入门代码 |
| **局限** | 只能看懂教学代码 |

### L2: 数据与内存模型
| 属性 | 内容 |
|:---|:---|
| **定义** | 值 vs 引用、栈 vs 堆、拷贝 vs 共享、指针/引用、可变/不可变 |
| **触发条件** | 理解 C/C++/Rust/Python 差距的根源 |
| **关键问题** | "我是否知道数据在哪？" |

### L3: 类型系统
| 属性 | 内容 |
|:---|:---|
| **定义** | 静态/动态类型、泛型/模板、类型约束、Null/Option |
| **触发条件** | 跨语言理解或类型错误调试 |
| **关键能力** | 一眼看穿泛型约束 |

### L4: 执行模型
| 属性 | 内容 |
|:---|:---|
| **定义** | 同步 vs 异步、阻塞 vs 非阻塞、线程 vs 协程、事件循环、内存可见性 |
| **触发条件** | 异步编程或并发问题 |
| **关键问题** | "我知道它什么时候执行、谁在等谁吗？" |

### L5: 错误模型
| 属性 | 内容 |
|:---|:---|
| **定义** | 异常 vs 返回值、panic/throw、RAII、defer/finally |
| **触发条件** | 资源泄漏或崩溃问题 |
| **关键问题** | "我知道失败会发生什么吗？" |

### L6: 元语法
| 属性 | 内容 |
|:---|:---|
| **定义** | 宏、装饰器、注解、反射、代码生成 |
| **触发条件** | 遇到"不像代码的代码" |
| **关键问题** | "我知道它在改写什么代码吗？" |

### L7: 范式
| 属性 | 内容 |
|:---|:---|
| **定义** | OOP、FP、过程式、声明式 |
| **触发条件** | 理解不同编程风格 |
| **关键问题** | "我知道这是对集合做变换，不是循环吗？" |

### L8: 领域语法
| 属性 | 内容 |
|:---|:---|
| **定义** | SQL、正则、Shell、DSL、框架约定 |
| **触发条件** | 理解真实项目代码 |
| **关键问题** | "我知道这不是编程吧？" |

### L9: 时间维度模型
| 属性 | 内容 |
|:---|:---|
| **定义** | 一次性计算 vs 多次复用、延迟执行、重复执行 |
| **触发条件** | 性能/Bug/竞态/重复执行问题 |
| **关键问题** | "我知道它什么时候跑、跑多久、是否重复跑吗？" |

### L10: 资源模型
| 属性 | 内容 |
|:---|:---|
| **定义** | CPU 密集 vs IO 密集 vs 内存绑定 vs 网络阻塞 |
| **触发条件** | 性能优化和并发模型选择 |
| **关键问题** | "数据在哪？处理是算还是等？能并行吗？" |

### L11: 隐含契约
| 属性 | 内容 |
|:---|:---|
| **定义** | 函数是否允许返回 None、是否允许 panic、是否线程安全等非语法规则 |
| **触发条件** | 生产级代码问题 |
| **关键问题** | "我知道作者在防什么吗？" |

### L12: 设计意图
| 属性 | 内容 |
|:---|:---|
| **定义** | 理解代码背后的设计决策（防 bug/防误用/性能换可读性等） |
| **触发条件** | 代码审查、架构设计、API 设计 |
| **关键问题** | "作者为什么要这么写？" |

## 四、项目架构概念

### Dataset First

| 属性 | 内容 |
|:---|:---|
| **定义** | 以 dataset 为边界组织代码开发、调度、部署、运维 |
| **触发条件** | 数据采集服务设计 |
| **核心文件** | contract.py, registry.py, collect.py, writer.py |
| **行为指引** | 先定义 dataset → 再实现 collect/backfill/repair/write/validate |

### Schema/Contract First

| 属性 | 内容 |
|:---|:---|
| **定义** | 先定义目标落表、字段语义、主键/幂等键、时间列，再写代码 |
| **触发条件** | 新建数据服务或重构数据模型 |
| **禁止** | 先跑代码再猜表结构 |

### 单一职责原则 (SRP)

| 属性 | 内容 |
|:---|:---|
| **定义** | 每个文件、每个类、每个函数只负责一件事 |
| **触发条件** | 代码混乱或职责不清时 |
| **判断标准** | 能用一句话描述这个模块吗？ |

### DRY 原则

| 属性 | 内容 |
|:---|:---|
| **定义** | Don't Repeat Yourself，不要重复代码 |
| **触发条件** | 发现重复代码时 |
| **行为指引** | 提取公共函数/类/模块复用 |

### KISS 原则

| 属性 | 内容 |
|:---|:---|
| **定义** | Keep It Simple, Stupid，保持简单 |
| **触发条件** | 过度设计或复杂度过高时 |
| **行为指引** | 减少复杂度、魔法代码、晦涩技巧 |

## 五、协作模式概念

### AI 蜂群协作

| 属性 | 内容 |
|:---|:---|
| **定义** | 利用 tmux 让多个 AI Agent 互相感知、通讯、控制 |
| **核心能力** | 感知（capture-pane）、控制（send-keys）、协调（共享状态文件） |
| **触发条件** | 多任务并行、需要 AI 互相监控救援 |
| **协作规则** | 先查后做、避免冲突、主动救援、状态广播 |

### Canvas 白板驱动开发

| 属性 | 内容 |
|:---|:---|
| **定义** | 让白板成为单一真相源，代码是白板的序列化形式 |
| **核心理念** | 图形是第一公民 |
| **触发条件** | 复杂架构设计、多人协作 |
| **工具** | Obsidian Canvas |

### 主从模式 (Master-Worker)

| 属性 | 内容 |
|:---|:---|
| **定义** | 一个指挥官 AI + 多个执行 Worker AI |
| **适用场景** | 复杂项目、需要统一协调 |
| **指挥官职责** | 全局监控、任务分配、冲突解决、故障救援 |

## 六、问题诊断概念

### 问题 = 目标 - 现状

| 属性 | 内容 |
|:---|:---|
| **定义** | 问题由差距产生，差距 = 目标状态 - 当前状态 |
| **分析要素** | 目标、现状、差距、判断标准 |
| **行为指引** | 先定义问题，再设计求解路径 |

### 问题求解 = 定义问题 × 构造解法 × 验证结果

| 属性 | 内容 |
|:---|:---|
| **定义** | 三要素循环 |
| **定义问题** | 目标、现状、差距 |
| **构造解法** | 对象、约束、路径 |
| **验证结果** | 反馈、迭代、收敛 |

## 八、提示词类型

### 元提示词 (Meta Prompt)

| 属性 | 内容 |

| 属性 | 内容 |
