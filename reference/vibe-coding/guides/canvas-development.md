# Canvas 白板驱动开发 (Canvas-Driven Development)

> **方法类型**：AI 协作范式
> **核心理念**：图形是第一公民，代码是白板的序列化形式
> **AI 执行要点**：让白板成为单一真相源

---

## 一、核心发现

```
传统开发流程：
├─ 写代码
├─ 口头沟通
├─ 脑补架构
├─ 代码失控
└─ 重构崩溃

Canvas 方式：
├─ 代码 ⇄ Canvas白板 ⇄ AI ⇄ 人类
└─ 单一事实来源
```

---

## 二、解决的问题

| 痛点 | 传统方式 | Canvas 方式 |
|:---|:---|:---|
| AI 看不懂项目结构 | 反复解释 | AI 直接看白板，秒懂架构 |
| 人类记不住复杂依赖 | 改 A 忘 B | 白板连线清晰，牵一发动全身 |
| 团队协作靠嘴说 | 翻代码猜 | 指着白板讲，新人 5 分钟看懂 |

---

## 三、工作流

### 3.1 写代码时自动更新白板

```
IF AI 生成代码 THEN
  ├─► 检测数据库写入 → 更新连接线
  ├─► 检测外部 API 调用 → 更新连接线
  └─► 检测模块依赖 → 更新关系图
END IF
```

### 3.2 人类和 AI 共同编辑白板

```
IF 人类在白板上操作 THEN
  ├─► 把 UserService 连线到 PaymentService
  └─► AI 理解意图后生成代码
END IF

IF AI 读懂意图 THEN
  ├─► 自动添加 import 语句
  ├─► 自动添加调用代码
  └─► 更新白板连接
END IF
```

### 3.3 白板成为开发中枢

| 操作 | 传统方式 | Canvas 方式 |
|:---|:---|:---|
| 要求 AI 重构 | "把支付逻辑拆出来" | 在白板拖出新节点 |
| Code Review | 逐行读代码 | 看白板连线 |
| 需求变更 | 到处改代码 | 白板删条线，AI 同步修改 |

## 四、关键创新

### 4.1 图形是第一公民

```
传统思维：
└─ 代码 → 文档（过期） → 架构图（更过期）

新思维：
└─ Canvas白板 = 唯一真相源 → 代码只是它的序列化形式
```

### 4.2 共享工作区

```
人类：擅长高层设计，在白板拖拽模块
AI：擅长细节实现，根据白板连线生成代码
协作方式：都编辑同一个白板
```

### 4.3 实时双向同步

```
代码变化 ──自动扫描──► 更新白板
白板编辑 ──AI解析──► 生成/修改代码
```

## 五、使用场景

### 5.1 给 AI 派活

```
传统：
"帮我写个用户注册功能，要连数据库，发邮件，记日志"

Canvas 方式：
1. 在白板画 3 个框：
   - RegisterAPI
   - Database
   - EmailService
2. 连线：RegisterAPI → Database/EmailService
3. 告诉 AI："按这个图实现"
```

### 5.2 Code Review

```
传统：逐行看代码，看晕了

Canvas 方式：
1. 看白板：发现前端直接连数据库
2. 拖动节点调整架构
3. AI 自动重构代码
```

### 5.3 接手新项目

```
传统：看 3 天代码还没懂

Canvas 方式：
1. 运行自动分析 → 1 分钟得到架构白板
2. 点开感兴趣的模块看详情
3. 直接在白板上画出要改的部分
```
