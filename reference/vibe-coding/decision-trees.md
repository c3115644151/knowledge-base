# Vibe Coding 决策树汇总 (Decision Trees)

> **文档类型**：条件→分支→动作 决策流程图
> **用途**：AI 在遇到特定情况时快速确定执行路径
> **格式**：Markdown 伪代码决策树

---

## 一、任务启动决策树

### 1.1 新项目启动

```
FUNCTION start_new_project()
  │
  ├─► IF 已有项目上下文 (memory-bank 存在)
  │     └─► READ architecture.md
  │     └─► READ progress.md
  │     └─► READ tech-stack.md
  │     └─► 确定下一步任务
  │     └─► GOTO execute_task()
  │
  └─► IF 无项目上下文
        ├─► 询问人类：项目目标、技术栈偏好
        ├─► 生成 game-design-document.md
        ├─► 生成 tech-stack.md
        ├─► 生成 implementation-plan.md
        ├─► 创建 memory-bank/ 目录
        ├─► 初始化 progress.md (空)
        ├─► 初始化 architecture.md (空)
        └─► GOTO execute_task()

END FUNCTION
```

### 1.2 新任务接收

```
FUNCTION receive_task(人类需求)
  │
  ├─► IF 需求模糊/不完整
  │     └─► 执行现象学还原
  │           ├─► 现象是什么？
  │           ├─► 意图是什么？
  │           ├─► 情境/约束是什么？
  │           ├─► 边界是什么？
  │           └─► 输出 MRE
  │
  ├─► IF 需求清晰
  │     ├─► 生成实施计划 (3-5 步)
  │     ├─► 每步包含：指令 + 验证方式
  │     ├─► 展示给人类确认
  │     └─► 等待确认后执行
  │
  └─► IF 人类要求快速执行
        ├─► 生成最小计划 (1-3 步)
        ├─► 包含验证步骤
        └─► 执行并实时报告

END FUNCTION
```

## 二、代码生成决策树

### 2.1 功能实现决策

```
FUNCTION implement_feature(功能描述)
  │
  ├─► IF 功能可能已存在于成熟库
  │     ├─► 搜索 GitHub Topics
  │     ├─► 评估 Star 数和维护状态
  │     ├─► IF 存在成熟方案
  │     │     ├─► 使用胶水编程
  │     │     ├─► 读取官方文档
  │     │     ├─► 编写胶水层
  │     │     └─► 验证连接正确
  │     └─► IF 无成熟方案
  │           └─► GOTO write_new_code()
  │
  └─► IF 必须自研
        GOTO write_new_code()

FUNCTION write_new_code()
  │
  ├─► IF 代码复杂度 > 500 行
  │     ├─► 拆分为多个模块
  │     ├─► 按功能边界拆分
  │     └─► 定义模块间接口
  │
  ├─► 编写单元测试
  ├─► 运行测试验证
  ├─► 更新 architecture.md (如需)
  └─► 更新 progress.md

END FUNCTION
```

### 2.2 依赖选择决策

```
FUNCTION select_dependency(需求)
  │
  ├─► IF 标准库能满足
  │     └─► 使用标准库
  │
  ├─► IF 已有项目依赖能满足
  │     └─► 复用现有依赖
  │
  ├─► IF 需要引入新依赖
  │     ├─► 搜索 GitHub
  │     ├─► 评估标准：
  │     │     ├─► Star 数 > 10k
  │     │     ├─► 近期有更新
  │     │     ├─► 有持续维护
  │     │     ├─► 许可证兼容
  │     │     └─► 社区活跃
  │     ├─► IF 满足标准
  │     │     ├─► 添加到 requirements.txt
  │     │     └─► 编写胶水层
  │     └─► IF 不满足标准
  │           └─► 报告人类，等待决策
  │
  └─► IF 自研 vs 复用
        ├─► 评估成本差异
        ├─► IF 复用成本明显更低
        │     └─► 必须复用
        └─► IF 自研更优
              └─► 说明原因，等待确认

END FUNCTION
```

## 三、问题诊断决策树

### 3.1 代码不工作

```
FUNCTION diagnose_issue(问题描述)
  │
  ├─► IF 有明确错误信息
  │     ├─► 完整复制错误信息
  │     ├─► 分析错误类型：
  │     │     ├─► SyntaxError → 检查语法
  │     │     ├─► ImportError → 检查依赖/路径
  │     │     ├─► TypeError → 检查类型不匹配
  │     │     ├─► ValueError → 检查值范围/格式
  │     │     ├─► RuntimeError → 分析执行上下文
  │     │     └─► 其他 → 溯因推理
  │     ├─► 提供修复方案
  │     └─► 输出修复代码
  │
  ├─► IF 无错误但行为异常
  │     ├─► 询问：预期行为是什么？
  │     ├─► 询问：实际行为是什么？
  │     ├─► 提供最小复现示例模板
  │     └─► 等待人类提供更多信息
  │
  └─► IF 性能问题
        ├─► 询问：具体哪里慢？
        ├─► 提供性能分析建议
        └─► 建议添加 benchmark

END FUNCTION
```

### 3.2 AI 陷入循环

```
FUNCTION handle_loop()
  │
  ├─► IF AI 反复修改同一问题
  │     ├─► 换个思路描述问题
  │     ├─► 提供更多上下文
  │     ├─► 简化问题范围
  │     └─► IF 仍然循环
  │           └─► 开新对话
  │
  ├─► IF 上下文过长
  │     └─► 使用 /clear 或 /compact
  │
  └─► IF 忘记之前约定
        └─► 重新提供关键上下文

END FUNCTION
```

### 3.3 溯因推理流程

```
FUNCTION abductive_reasoning(症状列表)
  │
  ├─► 列出候选原因 (≥3)
  │     ├─► 原因 A：可能性评分
  │     ├─► 原因 B：可能性评分
  │     └─► 原因 C：可能性评分
  │
  ├─► 为每个原因设计区分性实验
  │     ├─► 实验 1：验证/排除原因 A
  │     ├─► 实验 2：验证/排除原因 B
  │     └─► 实验 3：验证/排除原因 C
  │
  ├─► 按信息增益排序实验
  │
  ├─► 执行最高优先级实验
  │
  └─► IF 找到原因
        ├─► 实施修复
        ├─► 验证修复
        └─► 更新文档
        ELSE
        └─► 继续下一个实验

END FUNCTION
```

## 四、代码审查决策树

### 4.1 提交前自检

```
FUNCTION pre_commit_check()
  │
  ├─► 运行测试
  │     ├─► IF 全部通过 → 继续
  │     └─► IF 有失败 → 修复或标注
  │
  ├─► 代码风格检查
  │     ├─► 运行 ruff/black
  │     └─► 修复格式问题
  │
  ├─► 复杂度检查
  │     ├─► IF 函数过长 → 拆分
  │     └─► IF 嵌套过深 → 重构
  │
  ├─► 依赖检查
  │     ├─► 确认新依赖已添加
  │     └─► 确认无未使用导入
  │
  └─► 提交信息
        ├─► 格式：[类型] 简短描述
        ├─► 类型：feat/fix/docs/refactor/test
        └─► 描述：具体做了什么

END FUNCTION
```

### 4.2 代码质量评估

```
FUNCTION evaluate_quality(代码片段)
  │
  ├─► IF 违反 SOLID 原则
  │     ├─► 标记违反项
  │     └─► 建议重构方向
  │
  ├─► IF 违反 DRY 原则
  │     ├─► 标记重复代码
  │     └─► 建议提取公共函数
  │
  ├─► IF 缺乏错误处理
  │     ├─► 标记风险点
  │     └─► 建议添加 try/catch
  │
  ├─► IF 缺乏边界检查
  │     ├─► 标记风险点
  │     └─► 建议添加验证
  │
  └─► IF 存在安全风险
        ├─► 标记安全问题
        └─► 说明风险和修复建议

END FUNCTION
```

## 五、架构设计决策树

### 5.1 新模块设计

```
FUNCTION design_module(模块描述)
  │
  ├─► 确定模块边界
  │     ├─► 模块负责什么？
  │     ├─► 模块不负责什么？
  │     └─► 模块的公开接口是什么？
  │
  ├─► 确定依赖关系
  │     ├─► 模块依赖哪些？
  │     └─► 哪些依赖模块？
  │
  ├─► IF 依赖关系复杂
  │     ├─► 考虑使用依赖注入
  │     └─► 考虑使用接口/抽象
  │
  ├─► 确定数据流
  │     ├─► 输入是什么？
  │     ├─► 处理逻辑是什么？
  │     └─► 输出是什么？
  │
  └─► 评估可测试性
        ├─► 能否独立测试？
        └─► 需要 mock 什么？

END FUNCTION
```

### 5.2 技术选型决策

```
FUNCTION select_technology(需求)
  │
  ├─► 评估约束条件
  │     ├─► 性能要求
  │     ├─► 团队技术栈
  │     ├─► 维护成本
  │     └─► 时间限制
  │
  ├─► IF 已有类似系统
  │     └─► 参考现有技术选型
  │
  ├─► 生成候选方案 (2-3 个)
  │     ├─► 方案 A：保守/简单
  │     ├─► 方案 B：平衡
  │     └─► 方案 C：激进/创新
  │
  ├─► IF 人类决策
  │     ├─► 展示方案对比
  │     ├─► 说明各方案优缺点
  │     └─► 等待人类选择
  │
  └─► IF AI 推荐
        └─► 选择平衡方案并说明原因

END FUNCTION
```

## 六、安全决策树

### 6.1 凭证处理

```
FUNCTION handle_credentials()
  │
  ├─► IF 需要使用密钥/Token
  │     ├─► 必须从环境变量读取
  │     ├─► 禁止硬编码
  │     └─► 更新 .env.example
  │
  ├─► IF 发现硬编码凭证
  │     ├─► 标记为安全问题
  │     ├─► 建议使用环境变量
  │     └─► 不得执行提交
  │
  └─► IF 需要共享凭证
        └─► 使用密钥管理服务
              或加密配置文件

END FUNCTION
```

### 6.2 输入验证

```
FUNCTION validate_input(用户输入)
  │
  ├─► IF 必填字段为空
  │     └─► 返回明确错误信息
  │
  ├─► IF 类型不匹配
  │     └─► 返回类型错误
  │
  ├─► IF 超出允许范围
  │     └─► 返回边界错误
  │
  ├─► IF 包含可疑字符
  │     ├─► 检查 SQL 注入
  │     ├─► 检查 XSS
  │     └─► 检查命令注入
  │
  └─► IF 全部验证通过
        └─► 继续处理

END FUNCTION
```

## 七、协作决策树

### 7.1 人类在环决策

```
FUNCTION human_in_the_loop(决策点)
  │
  ├─► IF 是关键架构决策
  │     └─► 暂停，等待人类确认
  │
  ├─► IF 是风险操作 (删除/覆盖)
  │     ├─► 列出操作影响
  │     └─► 等待人类确认
  │
  ├─► IF 是业务逻辑判断
  │     └─► 列出选项，等待人类选择
  │
  └─► IF 是纯技术实现
  │     └─► AI 自主决策
  │
  └─► IF 人类要求 AI 自主决策
        └─► 执行并报告结果

END FUNCTION
```

```
FUNCTION design_test_strategy(模块)
  │

```

```
  │
  ├─► 分析反馈类型

  │
  └─► 更新文档
        ├─► 更新 progress.md
        └─► 更新 architecture.md (如需)
```

## 十、快速决策速查表

| 场景 | 快速决策 |
|:---|:---|
| 新项目启动 | 读取 memory-bank → 生成计划 → 确认执行 |
| 功能实现 | 搜索库 → 复用 → 胶水层 |
| 代码问题 | 错误信息 → 溯因推理 → 修复 |
| 架构决策 | 列出选项 → 等待人类确认 |
| 安全问题 | 立即标记 → 不执行 → 报告 |
| 测试失败 | 修复或标注 → 不得跳过 |
| 复杂度过高 | 拆分 → 简化 → 重构 |
| AI 循环 | 换思路 → 简化问题 → 开新对话 |
