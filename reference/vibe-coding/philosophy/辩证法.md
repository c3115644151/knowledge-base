# 辩证法 - 正反合思维框架

## 核心概念定义

### 正题（Thesis）
- **定义**：初始方案、最直觉的实现路径
- **特征**：快速、顺滑、优先功能实现
- **目的**：尽快跑通主路径，验证可行性

### 反题（Antithesis）
- **定义**：批判性审视、主动寻找缺陷
- **职责**：
  - 列举失败模式
  - 检查边界条件
  - 评估性能与安全隐患
  - 识别依赖风险
- **目的**：打破顺滑假设，暴露真实问题

### 合题（Synthesis）
- **定义**：整合正反两面，形成更优方案
- **原则**：速度与约束的平衡
- **产出**：重构后的接口、收敛的依赖、完整的测试

---

## 触发场景

| 场景 | 触发条件 | 执行重点 |
|:-----|:---------|:---------|
| 初次实现后 | 功能已跑通 | 立即进入"反"阶段 |
| 功能迭代时 | 引入新变更 | 检验对现有"合"的影响 |
| Bug修复时 | 定位问题根因 | 分析导致问题的"正"假设 |
| 重构前 | 准备大规模修改 | 验证当前"合"是否稳定 |
| 代码评审 | 接收外部反馈 | 将评审意见转化为"反"输入 |

## 执行协议

### Protocol: 正反合循环

```
┌─────────────────────────────────────────────────────────┐
│  阶段 1: 正（THESIS）- 快速实现                          │
├─────────────────────────────────────────────────────────┤
│  1.1 理解需求，明确核心路径                              │
│  1.2 生成"最顺"的实现方案                               │
│  1.3 确保最小可运行版本（MVP）                          │
│  1.4 验证基本功能可用                                    │
│                                                         │
│  产出：可运行的初始方案                                  │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  阶段 2: 反（ANTITHESIS）- 批判审计                      │
├─────────────────────────────────────────────────────────┤
│  2.1 列举所有失败模式                                    │
│  2.2 检查边界条件与异常输入                             │
│  2.3 评估性能瓶颈（时间/空间复杂度）                     │
│  2.4 识别安全风险与依赖隐患                              │
│  2.5 用工具落地验证：                                    │
│      □ 单元测试（覆盖率）                               │
│      □ 类型检查（TypeScript/类型注解）                  │
│      □ Lint（代码规范）                                 │
│      □ 基准测试（性能回归）                              │
│                                                         │
│  产出：问题清单与修复优先级                              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  阶段 3: 合（SYNTHESIS）- 整合收敛                        │
├─────────────────────────────────────────────────────────┤
│  3.1 优先级排序问题（关键/重要/次要）                    │
│  3.2 重构核心缺陷                                        │
│  3.3 收敛依赖，移除冗余                                  │
│  3.4 补齐测试与文档                                      │
│  3.5 验证整体稳定性                                      │
│                                                         │
│  产出：可用于下一轮"正"的稳定起点                       │
└─────────────────────────────────────────────────────────┘
```

### 简化口诀
```
正 → 顺写，先跑通
反 → 打脸，找问题
合 → 收敛，再出发
```

## AI行为决策指南

### 主动触发时机
作为 AI，在以下时刻主动触发正反合：

| 触发点 | AI行为 |
|:-------|:-------|
| 生成代码后 | 自动进入"反"阶段列举风险 |
| 用户说"可以了" | 主动提出"让我再检查一下" |
| 连续2次同类错误 | 触发"反"分析根因 |
| 方案过于简单 | 触发"反"检查复杂度是否足够 |

### 质量门禁
进入下一轮"正"前，必须通过：

```
□ 所有关键路径测试通过
□ 无高/中危Lint警告
□ 类型检查无错误
□ 文档与代码一致
□ 无已知的未处理边界情况
```

### 常见陷阱

| 陷阱 | 表现 | 纠正方法 |
|:-----|:-----|:---------|
| 正过载 | 一直写代码从不审计 | 强制每轮包含"反"阶段 |
| 反过载 | 过度分析无法收敛 | 设置问题数量上限，优先处理关键缺陷 |
| 合缺失 | 只修复不重构 | 要求重大修复必须伴随结构优化 |

## 与其他方法论的协同

### + 控制论
- 正反合是负反馈的具体实现形式
- "正"建立假设，"反"提供偏差信息，"合"执行修正

### + AI蜂群协作
- 主Agent执行"正"，审计Agent执行"反"
- 合题由协调Agent整合，形成下一轮分工
