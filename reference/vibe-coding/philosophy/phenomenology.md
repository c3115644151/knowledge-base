# 现象学还原方法 (Phenomenological Reduction)

> **方法类型**：问题分析协议
> **核心原则**：悬置假设，回到可观察事实
> **AI 执行触发**：需求含糊、模型脑补、Bug 难复现

---

## 一、方法定义

```
IF 遇到不清晰的问题 THEN
  → 不要推测原因
  → 先回到可观察的事实
  → 输出四件套：现象/意图/情境/边界
  → 再给最小修复与测试
END IF
```

---

## 二、四件套输出格式

### 2.1 现象（实际发生了什么）

| 属性 | 内容 |
|:---|:---|
| **定义** | 客观描述实际观察到的情况 |
| **格式** | 具体描述，不解释原因 |
| **示例** | "运行 python script.py 后，终端输出 'Connection timeout'" |

```
输出格式：
现象：[具体描述实际观察到的情况]
```

### 2.2 意图（期望发生什么）

| 属性 | 内容 |
|:---|:---|
| **定义** | 用户期望的正确行为 |
| **格式** | 具体描述期望结果 |
| **示例** | "期望脚本能成功连接数据库并返回查询结果" |

```
输出格式：
意图：[具体描述期望的行为]
```

### 2.3 情境（环境约束）

| 属性 | 内容 |
|:---|:---|
| **定义** | 问题发生的环境条件 |
| **包含** | 操作系统、版本、环境变量、网络状态等 |
| **示例** | "Python 3.9, macOS 13.2, 网络需要代理" |

```
输出格式：
情境：
- 操作系统：
- 运行时版本：
- 网络环境：
- 相关依赖版本：
- 其他约束：
```

### 2.4 边界（明确不做）

| 属性 | 内容 |
|:---|:---|
| **定义** | 明确排除的范围 |
| **格式** | 列出不在本次解决范围内的事项 |
| **示例** | "不涉及多线程问题、不涉及跨平台兼容" |

```
输出格式：
边界（明确不做）：
- [排除项 1]
- [排除项 2]
```

## 三、未确定项

| 属性 | 内容 |
|:---|:---|
| **定义** | 暂未确定或需要进一步调查的事项 |
| **格式** | 列出需要补充的信息 |

```
输出格式：
未确定项：
- [待确认项 1]
- [待确认项 2]
```

---

## 四、最小可复现示例 (MRE)

### 4.1 MRE 格式

| 组件 | 内容 |
|:---|:---|
| **最小输入** | 导致问题的最小数据量 |
| **最小脚本** | 复现问题的最小代码 |
| **复现步骤** | 清晰的重现步骤 |
| **预期 vs 实际** | 期望结果 vs 实际结果 |

### 4.2 MRE 模板

```
## 最小可复现示例 (MRE)

### 最小输入
[导致问题的最小输入数据]

### 最小脚本
```python
[复现问题的最小代码]
```

### 复现步骤
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

### 预期 vs 实际
- **预期**：[期望结果]
- **实际**：[实际结果]
```

## 五、AI 执行流程

```
FUNCTION phenomenology_reduction(问题描述)
  │
  ├─► 询问：现象是什么？
  │     └─► 输出：具体观察到的情况
  │
  ├─► 询问：意图是什么？
  │     └─► 输出：期望的正确行为
  │
  ├─► 询问：情境是什么？
  │     └─► 输出：环境约束信息
  │
  ├─► 询问：边界是什么？
  │     └─► 输出：明确排除的范围
  │
  ├─► 询问：未确定项？
  │     └─► 输出：需要补充的信息
  │
  ├─► 生成 MRE（最小可复现示例）
  │     ├─► 最小输入
  │     ├─► 最小脚本
  │     ├─► 复现步骤
  │     └─► 预期 vs 实际
  │
  └─► 基于事实分析原因
        └─► 输出：最小修复方案 + 测试

END FUNCTION
```

## 六、使用场景

| 场景 | 应用方式 |
|:---|:---|
| Bug 报告不清晰 | 要求用户按四件套重新描述 |
| AI 理解偏差 | 用四件套重新明确问题 |
| 需求模糊 | 现象学还原 → 明确真正需求 |
| 设计争议 | 现象学还原 → 明确各方案差异 |
