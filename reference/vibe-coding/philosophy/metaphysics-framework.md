# 世界观框架：理解世界、描述变化、整理知识的一套较小框架

> **框架类型**：认知框架
> **核心概念**：对象、状态、快照、序列、过程、变换、同一、差异、关系
> **AI 执行要点**：理解这些概念如何映射到代码和系统

---

## 一、概念框架总览

```
存在层：对象 → 状态（有什么、如何存在）
    ↓
表征层：快照 → 序列（如何记录、如何组织）
    ↓
生成层：过程 → 变换（如何变化、机制是什么）
    ↓
判定层：同一 → 差异（什么不变、什么改变）
    ↓
结构层：关系（如何连接成系统）
```

---

## 二、AI 视角下的概念映射

### 2.1 对象 (Object)

| 属性 | 内容 |
|:---|:---|
| **哲学定义** | 被识别成相对稳定的单位，有边界和持续性 |
| **代码映射** | 类实例、模块、进程、节点、数据实体 |
| **AI 执行触发** | 当需要识别系统边界和职责时 |

```
IF 需要分析系统结构 THEN
  → 识别核心对象（模块、类、服务）
  → 确定对象边界（接口、公开方法）
  → 确定对象标识（ID、名称、引用）
END IF
```

### 2.2 状态 (State)

| 属性 | 内容 |
|:---|:---|
| **哲学定义** | 对象在某个时刻的规定性总和 |
| **代码映射** | 内存值、配置、上下文、变量值 |
| **AI 执行触发** | 当需要理解系统当前行为时 |

```
IF 需要理解系统行为 THEN
  → 识别关键状态变量
  → 确定状态转换条件
  → 标注状态的不变量
END IF
```

### 2.3 快照 (Snapshot)

| 属性 | 内容 |
|:---|:---|
| **哲学定义** | 对状态的静态截取，强调"这一刻就是这样" |
| **代码映射** | 系统镜像、数据库备份、版本存档、日志点 |
| **AI 执行触发** | 当需要记录或恢复系统状态时 |

```
IF 需要记录系统状态 THEN
  → 选择合适的快照粒度
  → 确定快照内容（记录什么、忽略什么）
  → 标注快照的选择性
END IF
```

### 2.4 序列 (Sequence)

| 属性 | 内容 |
|:---|:---|
| **哲学定义** | 多个快照按时间或逻辑排列的结果 |
| **代码映射** | 日志、执行轨迹、输入流、事件链 |
| **AI 执行触发** | 当需要追踪系统变化历史时 |

```
IF 需要追踪变化历史 THEN
  → 确定序列的排序依据（时间/逻辑/因果）
  → 识别序列中的关键节点
  → 分析序列的连续性和断点
END IF
```

### 2.5 过程 (Process)

| 属性 | 内容 |
|:---|
| **哲学定义** | 序列的动态整体，有持续性、方向性、内在机制 |
| **代码映射** | 程序运行、工作流、协议执行、数据处理管道 |
| **AI 执行触发** | 当需要设计或优化系统流程时 |

```
IF 需要设计系统流程 THEN
  → 识别过程的起点和终点
  → 确定过程的展开方向
  → 分析过程中的状态转换
  → 识别过程中的关键机制
END IF
```

### 2.6 变换 (Transformation)

| 属性 | 内容 |
|:---|:---|
| **哲学定义** | 从一个状态到另一个状态的规则、操作或机制 |
| **代码映射** | 函数、算法、状态转移函数、数据处理规则 |
| **AI 执行触发** | 当需要理解或实现状态变化时 |

```
IF 需要实现状态变化 THEN
  → 明确输入状态和输出状态
  → 确定变换规则（确定性/概率性）
  → 验证变换的正确性
  → 分析变换的可逆性
END IF
```

## 三、判定层概念

### 3.1 同一 (Identity)

| 属性 | 内容 |
|:---|:---|
| **哲学定义** | 在变化中，某个东西依然被认作"它自己" |
| **代码映射** | 对象 ID、引用、版本继承、持续性标识 |
| **AI 执行触发** | 当需要判断对象是否仍是同一个时 |

```
同一性判断标准：
├─ 物质连续性：组件是否相同
├─ 结构连续性：结构是否保持
├─ 功能连续性：功能是否保持
├─ 因果连续性：因果链是否保持
└─ 规则连续性：规则是否保持
```

### 3.2 差异 (Difference)

| 属性 | 内容 |
|:---|:---|
| **哲学定义** | 对象之间、状态之间、过程之间的不相同 |
| **代码映射** | 补丁、变更记录、版本比较、条件分支 |
| **AI 执行触发** | 当需要识别变化或进行比较时 |

```
IF 需要识别变化 THEN
  → 比较当前状态与期望状态
  → 生成差异报告
  → 分析差异的语义意义
END IF
```

## 四、结构层概念

### 4.1 关系 (Relation)

| 属性 | 内容 |
|:---|:---|
| **哲学定义** | 对象和对象、状态和状态之间的连接方式 |
| **代码映射** | 依赖、调用、连接、图结构、API 调用 |
| **AI 执行触发** | 当需要理解系统整体结构时 |

```
IF 需要理解系统结构 THEN
  → 识别对象间的关系类型（调用/包含/引用）
  → 确定关系的方向（单向/双向）
  → 分析关系的性质（同步/异步、紧耦合/松耦合）
  → 绘制关系图（依赖图、调用图）
END IF
```

## 五、AI 执行框架

### 5.1 系统分析框架

```
面对任何复杂系统，按以下步骤分析：

1. 确定对象（对象层）
   └─► 识别核心模块/类/服务
   └─► 确定对象边界和标识

2. 描述状态（状态层）
   └─► 识别关键状态变量
   └─► 确定状态的合法范围

3. 收集快照（快照层）
   └─► 在关键节点记录状态
   └─► 确定快照的粒度

4. 排列序列（序列层）
   └─► 按时间/逻辑排序快照
   └─► 识别关键序列模式

5. 识别过程（过程层）
   └─► 从序列中识别过程
   └─► 确定过程的起点和终点

6. 分析变换（变换层）
   └─► 确定推动变化的核心机制
   └─► 分析变换的可逆性

7. 判断同一（同一层）
   └─► 确定同一性判断标准
   └─► 追踪对象的持续性

8. 识别差异（差异层）
   └─► 比较不同版本/状态
   └─► 分析差异的语义

9. 理解关系（关系层）
   └─► 绘制依赖/调用关系图
   └─► 确定关系类型和方向
```

### 5.2 代码组织框架

| 概念层 | 代码组织对应 |
|:---|:---|
| 对象 | 模块、类、函数 |
| 状态 | 变量、配置、上下文 |
| 快照 | 备份、版本、日志点 |
| 序列 | 执行轨迹、事件流 |
| 过程 | 工作流、管道、处理流程 |
| 变换 | 函数、算法、转换器 |
| 同一 | ID、引用、版本 |
| 差异 | diff、patch、变更 |
| 关系 | 依赖、调用、导入 |

## 六、跨学科应用

### 6.1 数据库系统

```
对象 → 表/集合
状态 → 行数据/字段值
快照 → 备份/存档
序列 → 事务日志/binlog
过程 → 查询处理流程
变换 → SQL 操作
同一 → 主键/唯一标识
差异 → 更新前后的 diff
关系 → 外键/ JOIN
```

### 6.2 分布式系统

```
对象 → 服务/节点/进程
状态 → 内存状态/配置
快照 → checkpoint/快照
序列 → 事件日志/消息队列
过程 → 工作流/编排
变换 → 状态转移函数
同一 → 分布式 ID/一致性哈希
差异 → 版本向量/冲突检测
关系 → 网络拓扑/服务发现
```

### 6.3 API 系统

```
对象 → 资源/端点
状态 → 请求上下文/会话
快照 → 请求/响应记录
序列 → API 调用链
过程 → 请求处理流程
变换 → 请求 → 响应
同一 → 请求 ID/幂等键
差异 → 请求差异分析
关系 → 依赖 API/调用方
```

## 七、实践要点

### 7.1 设计原则

```
IF 设计新系统 THEN
  → 先确定核心对象和边界
  → 定义清晰的状态空间
  → 设计状态变换机制
  → 确定同一性判断标准
  → 明确对象间关系
END IF
```

### 7.2 问题诊断原则

```
IF 系统出现问题 THEN
  → 识别当前状态
  → 追溯状态变化序列
  → 分析变换机制
  → 检查对象间关系
  → 确定同一性是否被破坏
END IF
```
